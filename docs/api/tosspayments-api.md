# 토스페이먼츠 개발용 API 문서

확인일: 2026-07-07

원본:

- 개발자센터: `https://developers.tosspayments.com/`
- 결제위젯 연동 가이드: `https://docs.tosspayments.com/guides/v2/payment-widget/integration`
- API/SDK 문서: `https://docs.tosspayments.com/reference`

본 문서는 외부 문서 장애 또는 접근 지연에 대비한 개발용 요약이다. 실제 구현 전에는 원본 문서와 테스트 키로 요청/응답 필드를 재검증해야 한다. 클라이언트 키, 시크릿 키, 결제 테스트 키는 이 문서에 기록하지 않는다.

## 1. 1차 결제 방향

- 결제 UI는 토스페이먼츠 결제위젯을 사용한다.
- 결제수단은 결제위젯 기본 제공 수단을 사용한다.
- 실제 노출 결제수단은 토스페이먼츠 계약과 관리자 설정을 따른다.
- 쇼핑몰은 결제 전 주문을 만들고, 결제 승인 성공 후 주문을 확정한다.
- 결제 완료 후 사방넷 주문 전송을 시도한다.

## 2. 키 관리

| 키 | 사용 위치 | 저장 방식 |
| --- | --- | --- |
| 클라이언트 키 | 브라우저 결제위젯 초기화 | 공개 가능하나 환경변수로 관리 |
| 시크릿 키 | 서버 승인/조회/취소 API | 배포 시크릿 |
| 웹훅 시크릿 | 웹훅 검증 사용 시 | 배포 시크릿 |

주의:

- 시크릿 키는 프론트엔드 코드에 포함하지 않는다.
- 서버 로그에 시크릿 키, 인증 헤더, 전체 카드 정보가 남지 않게 한다.
- 테스트 키와 운영 키를 환경별로 분리한다.

## 3. 결제 흐름

### 기본 흐름

1. 사용자가 장바구니 또는 바로구매에서 주문서를 작성한다.
2. 서버가 상품 가격, 재고, 쿠폰, 적립금, 배송비를 계산한다.
3. 서버가 내부 주문을 `결제대기` 상태로 생성하고 `orderId`를 발급한다.
4. 클라이언트가 결제위젯을 렌더링한다.
5. 사용자가 결제수단을 선택하고 결제를 요청한다.
6. 토스페이먼츠가 성공 URL 또는 실패 URL로 사용자를 돌려보낸다.
7. 성공 URL에서 서버가 `paymentKey`, `orderId`, `amount`를 검증한다.
8. 서버가 결제 승인을 완료 또는 승인 결과를 확인한다.
9. 내부 주문을 `결제완료`로 변경하고 결제 기록을 저장한다.
10. 사방넷 주문 전송을 시도한다.
11. 사용자에게 주문 완료 화면을 보여준다.

### 실패 흐름

- 결제 실패 URL로 돌아오면 내부 주문은 `결제실패` 또는 `결제대기 만료` 상태로 둔다.
- 실패 사유 코드를 저장한다.
- 사용자는 주문서로 돌아가 다시 결제할 수 있어야 한다.
- 재결제 시 기존 결제대기 주문을 재사용할지 새 주문을 만들지는 구현 정책으로 정한다.

### 취소 흐름

- 사용자가 결제창을 닫거나 취소하면 주문은 결제 완료 처리하지 않는다.
- 결제 승인 이후 취소는 서버에서 토스페이먼츠 취소 API를 호출해야 한다.
- 취소 성공 후 내부 주문, 쿠폰, 적립금, 사방넷 상태를 함께 보정한다.

## 4. 서버 검증 요구사항

서버는 다음 값을 반드시 검증한다.

| 항목 | 검증 내용 |
| --- | --- |
| `orderId` | 내부 주문 존재 여부와 상태 |
| 결제 금액 | 서버 계산 최종 금액과 일치 여부 |
| `paymentKey` | 이미 처리된 결제인지 여부 |
| 주문 상태 | `결제대기` 상태인지 여부 |
| 사용자 | 회원 주문이면 주문 소유자, 비회원 주문이면 주문 세션 또는 조회 키 |
| 쿠폰/적립금 | 결제 직전 사용 가능 상태 유지 여부 |
| 재고/옵션 | 결제 직전 유효성 유지 여부 |

금액 불일치, 중복 승인, 이미 취소된 주문, 만료된 주문은 승인하지 않는다.

## 5. 내부 주문 상태 초안

| 상태 | 설명 |
| --- | --- |
| `draft` | 주문서 작성 중 |
| `payment_pending` | 결제대기 |
| `payment_failed` | 결제 실패 |
| `paid` | 결제 완료 |
| `sabangnet_pending` | 사방넷 전송 대기 |
| `sabangnet_failed` | 사방넷 전송 실패 |
| `order_confirmed` | 사방넷 전송 또는 운영 확인 완료 |
| `shipping_ready` | 배송 준비 |
| `shipping` | 배송 중 |
| `delivered` | 배송 완료 |
| `cancel_requested` | 취소 요청 |
| `cancelled` | 취소 완료 |
| `return_requested` | 반품 요청 |
| `returned` | 반품 완료 |

## 6. 결제 기록 저장 항목

| 항목 | 설명 |
| --- | --- |
| 내부 주문 ID | 쇼핑몰 주문 식별자 |
| `orderId` | 토스페이먼츠 주문 ID |
| `paymentKey` | 결제 식별자 |
| 결제 상태 | 승인, 실패, 취소 등 |
| 결제 금액 | 최종 결제 금액 |
| 결제수단 | 카드, 간편결제 등 응답 기준 |
| 승인 시각 | 결제 완료 시각 |
| 실패 코드 | 실패 시 저장 |
| 실패 메시지 | 사용자 표시용과 내부 로그용 구분 |
| 원본 응답 요약 | 민감정보 제외 저장 |

## 7. 웹훅/상태 보정

결제 결과가 브라우저 복귀만으로 처리되면 사용자가 중간에 이탈했을 때 상태 불일치가 생길 수 있다. 다음 중 하나 이상을 적용한다.

- 토스페이먼츠 웹훅을 받아 결제 상태를 보정한다.
- 관리자에서 결제 재조회 기능을 제공한다.
- 배치 작업으로 `결제대기` 주문 중 결제 완료 여부를 재확인한다.

권장:

- 1차 구현에는 성공 URL 기반 승인 처리와 관리자 재조회 기능을 포함한다.
- 웹훅은 가능하면 1차에 포함하되, 계약/개발 일정에 따라 우선순위를 조정한다.

## 8. 취소/환불 요구사항

- 전체 취소와 부분 취소 가능 여부는 토스페이먼츠 계약과 운영 정책으로 확정한다.
- 취소 요청은 내부 주문 상태, 배송 상태, 사방넷 상태를 확인한 뒤 처리한다.
- 결제 취소 성공 후 쿠폰 복구, 적립금 회수, 배송비 재계산을 수행한다.
- 취소 실패 시 내부 주문 상태를 임의로 취소 완료로 바꾸지 않는다.

## 9. 관리자 기능

관리자는 다음 결제 관련 기능을 사용할 수 있어야 한다.

- 주문별 결제 상태 조회
- 결제 원장 조회
- 결제 승인 결과 재조회
- 취소 요청과 취소 결과 확인
- 결제 실패 사유 확인
- 사방넷 전송 상태와 결제 상태 대조
- 수동 보정 로그 기록

## 10. 장애 대응

- 토스페이먼츠 API 장애 시 결제 승인 여부가 불명확한 주문은 `확인필요` 상태로 둔다.
- 사용자에게 중복 결제를 유도하지 않도록 결제 결과 확인 안내를 제공한다.
- API 타임아웃 후 같은 `orderId`로 중복 승인 요청하지 않도록 멱등 처리를 한다.
- 결제 성공 후 사방넷 전송 실패는 결제 취소로 자동 처리하지 않는다.
- 운영자가 결제 상태와 사방넷 전송 상태를 분리해서 확인할 수 있어야 한다.

## 11. 구현 전 체크리스트

- [ ] 결제위젯 샘플 결제 성공 확인
- [ ] 성공 URL, 실패 URL, 취소 URL 설계
- [ ] 서버 승인 검증 구현 범위 확정
- [ ] 웹훅 사용 여부 결정

- [ ] 비회원 주문의 결제 후 조회 방식 확정
- [ ] 사방넷 전송 실패 시 운영 알림 방식 확정
