# 토스페이먼츠 개발용 API 문서

확인일: 2026-07-07

원본:

- 개발자센터: `https://developers.tosspayments.com/`
- 결제위젯 연동 가이드: `https://docs.tosspayments.com/guides/v2/payment-widget/integration`
- API/SDK 문서: `https://docs.tosspayments.com/reference`

본 문서는 외부 문서 장애 또는 접근 지연에 대비한 개발용 요약이다. 실제 구현 전에는 원본 문서와 테스트 키로 요청/응답 필드를 재검증해야 한다. 클라이언트 키, 시크릿 키, 결제 테스트 키는 이 문서에 기록하지 않는다.

## 1. 1차 결제 방향

- 결제 UI는 토스페이먼츠 결제위젯을 사용한다.
- 결제수단은 결제위젯 기본 제공 수단을 사용한다.
- 실제 노출 결제수단은 토스페이먼츠 계약과 관리자 설정을 따른다.
- 쇼핑몰은 결제 전 주문을 만들고, 결제 승인 성공 후 주문을 확정한다.
- 결제 완료 후 사방넷 주문 전송을 시도한다.

## 2. 키 관리

| 키 | 사용 위치 | 저장 방식 |
| --- | --- | --- |
| 클라이언트 키 | 브라우저 결제위젯 초기화 | 공개 가능하나 환경변수로 관리 |
| 시크릿 키 | 서버 승인/조회/취소 API | 배포 시크릿 |
| 웹훅 시크릿 | 웹훅 검증 사용 시 | 배포 시크릿 |

주의:

- 시크릿 키는 프론트엔드 코드에 포함하지 않는다.
- 서버 로그에 시크릿 키, 인증 헤더, 전체 카드 정보가 남지 않게 한다.
- 테스트 키와 운영 키를 환경별로 분리한다.

## 3. 결제 흐름

### 기본 흐름

1. 사용자가 장바구니 또는 바로구매에서 주문서를 작성한다.
2. 서버가 상품 가격, 재고, 쿠폰, 적립금, 배송비를 계산한다.
3. 서버가 내부 주문을 `결제대기` 상태로 생성하고 `orderId`를 발급한다.
4. 클라이언트가 결제위젯을 렌더링한다.
5. 사용자가 결제수단을 선택하고 결제를 요청한다.
6. 토스페이먼츠가 성공 URL 또는 실패 URL로 사용자를 돌려보낸다.
7. 성공 URL에서 서버가 `paymentKey`, `orderId`, `amount`를 검증한다.
8. 서버가 결제 승인을 완료 또는 승인 결과를 확인한다.
9. 내부 주문을 `결제완료`로 변경하고 결제 기록을 저장한다.
10. 사방넷 주문 전송을 시도한다.
11. 사용자에게 주문 완료 화면을 보여준다.

### 실패 흐름

- 결제 실패 URL로 돌아오면 내부 주문은 `결제실패` 또는 `결제대기 만료` 상태로 둔다.
- 실패 사유 코드를 저장한다.
- 사용자는 주문서로 돌아가 다시 결제할 수 있어야 한다.
- 재결제 시 기존 결제대기 주문을 재사용할지 새 주문을 만들지는 구현 정책으로 정한다.

### 취소 흐름

- 사용자가 결제창을 닫거나 취소하면 주문은 결제 완료 처리하지 않는다.
- 결제 승인 이후 취소는 서버에서 토스페이먼츠 취소 API를 호출해야 한다.
- 취소 성공 후 내부 주문, 쿠폰, 적립금, 사방넷 상태를 함께 보정한다.

## 4. API 입출력 상세

토스페이먼츠 결제위젯 연동은 클라이언트 SDK 호출과 서버 API 호출이 분리된다. 클라이언트는 결제창을 열고, 서버는 최종 승인/조회/취소를 담당한다.

### 4.1 결제위젯 초기화

- 실행 위치: 브라우저
- 용도: 토스페이먼츠 결제위젯 SDK를 초기화한다.
- 입력 형태:

```json
{
  "clientKey": "환경변수에서 주입한 클라이언트 키",
  "customerKey": "회원 고유키 또는 비회원용 익명 고유키"
}
```

| 필드 | 설명 |
| --- | --- |
| `clientKey` | 토스페이먼츠 클라이언트 키. 프론트에서 사용하지만 환경별로 분리한다. |
| `customerKey` | 결제위젯에서 고객을 식별하는 값. 비회원은 주문 단위 임시 키를 사용한다. |

출력/저장:

- SDK 인스턴스.
- 서버에는 `clientKey`를 저장하지 않고 환경 설정으로만 관리한다.
- `customerKey`는 주문과 결제 로그에 남겨 재현 가능하게 한다.

### 4.2 결제위젯 렌더링

- 실행 위치: 브라우저
- 용도: 결제수단 영역과 약관 영역을 표시한다.
- 입력 형태:

```json
{
  "amount": {
    "currency": "KRW",
    "value": 59000
  },
  "variantKey": "DEFAULT"
}
```

| 필드 | 설명 |
| --- | --- |
| `amount.currency` | 통화. 국내 의류 쇼핑몰 1차는 `KRW` |
| `amount.value` | 서버가 계산한 최종 결제 금액 |
| `variantKey` | 토스페이먼츠 관리자에서 설정한 결제 UI variant |

주의:

- 금액은 서버 계산값을 사용한다.
- 쿠폰/적립금/배송비 변경 시 결제위젯 금액도 다시 맞춘다.

### 4.3 결제 요청

- 실행 위치: 브라우저
- 용도: 결제창을 열고 사용자를 토스페이먼츠 인증/결제 흐름으로 보낸다.
- 입력 형태:

```json
{
  "orderId": "SEQ-20260707-000001",
  "orderName": "반팔티셔츠 외 1건",
  "successUrl": "https://www.example.com/payments/toss/success",
  "failUrl": "https://www.example.com/payments/toss/fail",
  "customerEmail": "buyer@example.com",
  "customerName": "홍길동",
  "customerMobilePhone": "01012345678"
}
```

| 필드 | 설명 |
| --- | --- |
| `orderId` | 쇼핑몰이 생성한 주문번호. 6~64자, 영문/숫자/`-`/`_` 사용 |
| `orderName` | 구매상품명. 예: `상품명 외 N건` |
| `successUrl` | 결제 인증 성공 후 이동할 URL |
| `failUrl` | 결제 실패 후 이동할 URL |
| `customerEmail`, `customerName`, `customerMobilePhone` | 구매자 정보. 비회원 주문도 주문서 정보로 전달 |

성공 리다이렉트 출력:

```json
{
  "paymentKey": "토스페이먼츠 결제 키",
  "orderId": "SEQ-20260707-000001",
  "amount": 59000,
  "paymentType": "NORMAL"
}
```

실패 리다이렉트 출력:

```json
{
  "code": "PAY_PROCESS_CANCELED",
  "message": "사용자가 결제를 취소했습니다.",
  "orderId": "SEQ-20260707-000001"
}
```

주의:

- 성공 URL의 `amount`는 결제 요청 시 서버가 계산한 금액과 반드시 일치해야 한다.
- `paymentKey`, `orderId`, `amount`는 서버 승인 요청 전 임시 저장한다.
- 실패 시 `orderId`가 없을 수 있으므로 세션/주문 컨텍스트로도 추적한다.

### 4.4 결제 승인

- 실행 위치: 서버
- Method/Path: `POST /v1/payments/confirm`
- 인증: Secret Key 기반 Basic 인증
- 용도: 성공 리다이렉트로 받은 결제 건을 최종 승인한다.
- 입력 형태:

```json
{
  "paymentKey": "토스페이먼츠 결제 키",
  "orderId": "SEQ-20260707-000001",
  "amount": 59000
}
```

| 필드 | 설명 |
| --- | --- |
| `paymentKey` | 결제 식별자. 승인, 조회, 취소에 계속 사용하므로 저장 필수 |
| `orderId` | 결제 요청 시 보낸 내부 주문번호 |
| `amount` | 최종 승인 금액 |

성공 응답 형태: `Payment` 객체

```json
{
  "paymentKey": "토스페이먼츠 결제 키",
  "type": "NORMAL",
  "orderId": "SEQ-20260707-000001",
  "orderName": "반팔티셔츠 외 1건",
  "method": "카드",
  "totalAmount": 59000,
  "balanceAmount": 59000,
  "status": "DONE",
  "requestedAt": "2026-07-07T10:00:00+09:00",
  "approvedAt": "2026-07-07T10:01:00+09:00",
  "card": {},
  "easyPay": {},
  "virtualAccount": null,
  "cancels": null
}
```

저장 필드:

| 필드 | 내부 용도 |
| --- | --- |
| `paymentKey` | 결제 조회/취소 키 |
| `type` | `NORMAL`, `BRANDPAY`, `KEYIN` 중 의도한 결제 타입 검증 |
| `orderId` | 내부 주문 매칭 |
| `method` | 결제수단 표시 |
| `totalAmount`, `balanceAmount` | 결제 금액/취소 가능 금액 |
| `status` | `DONE`, `CANCELED`, `PARTIAL_CANCELED` 등 상태 |
| `requestedAt`, `approvedAt` | 결제 요청/승인 시각 |
| `card`, `easyPay`, `transfer`, `virtualAccount` | 결제수단별 상세. 민감정보 제외 저장 |
| `cancels[]` | 취소 이력 |

실패 응답 형태:

```json
{
  "code": "ERROR_CODE",
  "message": "실패 사유"
}
```

처리 규칙:

- 승인 전 내부 주문 상태가 `payment_pending`인지 확인한다.
- 서버 계산 금액과 `amount`가 다르면 승인하지 않는다.
- 동일 `paymentKey` 또는 `orderId`가 이미 승인됐으면 중복 처리하지 않는다.
- 승인 성공 후 내부 주문을 `paid`로 바꾸고 사방넷 전송을 시도한다.

### 4.5 paymentKey로 결제 조회

- 실행 위치: 서버 또는 관리자 도구
- Method/Path: `GET /v1/payments/{paymentKey}`
- 인증: Secret Key 기반 Basic 인증
- 용도: 결제 상태 불일치 보정, 관리자 재조회, 취소 전 상태 확인.
- 입력 위치: path

```json
{
  "paymentKey": "토스페이먼츠 결제 키"
}
```

성공 응답: `Payment` 객체.

저장/보정:

- `status`, `balanceAmount`, `cancels[]`, `approvedAt`를 내부 결제 원장과 대조한다.
- 내부 주문이 결제대기인데 토스 상태가 `DONE`이면 관리자 확인 후 결제완료로 보정한다.
- 내부 주문이 결제완료인데 토스 상태가 `CANCELED` 또는 `PARTIAL_CANCELED`이면 취소/환불 이력을 보정한다.

### 4.6 orderId로 결제 조회

- 실행 위치: 서버 또는 관리자 도구
- Method/Path: `GET /v1/payments/orders/{orderId}`
- 인증: Secret Key 기반 Basic 인증
- 용도: 성공 리다이렉트 유실 등 `paymentKey` 저장이 누락된 주문을 내부 주문번호로 조회한다.
- 입력 위치: path

```json
{
  "orderId": "SEQ-20260707-000001"
}
```

성공 응답: `Payment` 객체.

### 4.7 결제 취소

- 실행 위치: 서버 또는 관리자 도구
- Method/Path: `POST /v1/payments/{paymentKey}/cancel`
- 인증: Secret Key 기반 Basic 인증
- 권장 헤더: 멱등키
- 용도: 결제 완료 주문의 전체 취소 또는 부분 취소.
- 입력 위치: path + JSON body

전체 취소 입력:

```json
{
  "cancelReason": "고객 요청"
}
```

부분 취소 입력:

```json
{
  "cancelReason": "부분 반품",
  "cancelAmount": 20000,
  "taxFreeAmount": 0,
  "currency": "KRW"
}
```

가상계좌 환불 계좌가 필요한 경우:

```json
{
  "cancelReason": "고객 요청",
  "cancelAmount": 20000,
  "refundReceiveAccount": {
    "bank": "은행 코드",
    "accountNumber": "하이픈 없는 계좌번호",
    "holderName": "예금주"
  }
}
```

성공 응답: `Payment` 객체. `cancels[]`에 취소 객체가 추가된다.

취소 객체 저장 후보:

| 필드 | 설명 |
| --- | --- |
| `transactionKey` | 취소 거래 식별자 |
| `cancelReason` | 취소 사유 |
| `cancelAmount` | 취소 금액 |
| `taxFreeAmount` | 면세 취소 금액 |
| `refundableAmount` | 취소 후 환불 가능 잔액 |
| `canceledAt` | 취소 시각 |

처리 규칙:

- 배송 상태, 사방넷 상태, 쿠폰/적립금 회수 가능 여부를 먼저 확인한다.
- `cancelAmount`가 없으면 전체 취소다.
- 부분 취소 사용 여부는 토스페이먼츠 계약과 운영 정책 확정 후 활성화한다.
- 같은 취소 요청이 중복 실행되지 않도록 멱등키와 내부 취소 요청 ID를 사용한다.

### 4.8 웹훅

- 실행 위치: 서버
- 용도: 브라우저 이탈, 네트워크 장애, 비동기 결제수단 상태 변경을 보정한다.
- 입력 형태: 토스페이먼츠 이벤트 본문. 이벤트 종류와 본문 필드는 원본 문서에서 최종 확인한다.

저장 필드 후보:

| 필드 | 설명 |
| --- | --- |
| `eventType` | 이벤트 종류 |
| `createdAt` | 이벤트 생성 시각 |
| `paymentKey` | 결제 식별자 |
| `orderId` | 주문번호 |
| `status` | 결제 상태 |
| `rawPayload` | 민감정보 제외 원본 요약 |

처리 규칙:

- 웹훅 서명을 검증할 수 있으면 반드시 검증한다.
- 같은 이벤트가 재전송될 수 있으므로 이벤트 ID 또는 본문 해시로 중복 처리한다.
- 웹훅만 믿고 주문을 확정하지 않고, 필요 시 결제 조회 API로 재확인한다.

## 5. 서버 검증 요구사항

서버는 다음 값을 반드시 검증한다.

| 항목 | 검증 내용 |
| --- | --- |
| `orderId` | 내부 주문 존재 여부와 상태 |
| 결제 금액 | 서버 계산 최종 금액과 일치 여부 |
| `paymentKey` | 이미 처리된 결제인지 여부 |
| 주문 상태 | `결제대기` 상태인지 여부 |
| 사용자 | 회원 주문이면 주문 소유자, 비회원 주문이면 주문 세션 또는 조회 키 |
| 쿠폰/적립금 | 결제 직전 사용 가능 상태 유지 여부 |
| 재고/옵션 | 결제 직전 유효성 유지 여부 |

금액 불일치, 중복 승인, 이미 취소된 주문, 만료된 주문은 승인하지 않는다.

## 6. 내부 주문 상태 초안

| 상태 | 설명 |
| --- | --- |
| `draft` | 주문서 작성 중 |
| `payment_pending` | 결제대기 |
| `payment_failed` | 결제 실패 |
| `paid` | 결제 완료 |
| `sabangnet_pending` | 사방넷 전송 대기 |
| `sabangnet_failed` | 사방넷 전송 실패 |
| `order_confirmed` | 사방넷 전송 또는 운영 확인 완료 |
| `shipping_ready` | 배송 준비 |
| `shipping` | 배송 중 |
| `delivered` | 배송 완료 |
| `cancel_requested` | 취소 요청 |
| `cancelled` | 취소 완료 |
| `return_requested` | 반품 요청 |
| `returned` | 반품 완료 |

## 8. 웹훅/상태 보정

결제 결과가 브라우저 복귀만으로 처리되면 사용자가 중간에 이탈했을 때 상태 불일치가 생길 수 있다. 다음 중 하나 이상을 적용한다.

- 토스페이먼츠 웹훅을 받아 결제 상태를 보정한다.
- 관리자에서 결제 재조회 기능을 제공한다.
- 배치 작업으로 `결제대기` 주문 중 결제 완료 여부를 재확인한다.

권장:

- 1차 구현에는 성공 URL 기반 승인 처리와 관리자 재조회 기능을 포함한다.
- 웹훅은 가능하면 1차에 포함하되, 계약/개발 일정에 따라 우선순위를 조정한다.

## 9. 취소/환불 요구사항

- 전체 취소와 부분 취소 가능 여부는 토스페이먼츠 계약과 운영 정책으로 확정한다.
- 취소 요청은 내부 주문 상태, 배송 상태, 사방넷 상태를 확인한 뒤 처리한다.
- 결제 취소 성공 후 쿠폰 복구, 적립금 회수, 배송비 재계산을 수행한다.
- 취소 실패 시 내부 주문 상태를 임의로 취소 완료로 바꾸지 않는다.

## 10. 관리자 기능

관리자는 다음 결제 관련 기능을 사용할 수 있어야 한다.

- 주문별 결제 상태 조회
- 결제 원장 조회
- 결제 승인 결과 재조회
- 취소 요청과 취소 결과 확인
- 결제 실패 사유 확인
- 사방넷 전송 상태와 결제 상태 대조
- 수동 보정 로그 기록

## 11. 장애 대응

- 토스페이먼츠 API 장애 시 결제 승인 여부가 불명확한 주문은 `확인필요` 상태로 둔다.
- 사용자에게 중복 결제를 유도하지 않도록 결제 결과 확인 안내를 제공한다.
- API 타임아웃 후 같은 `orderId`로 중복 승인 요청하지 않도록 멱등 처리를 한다.
- 결제 성공 후 사방넷 전송 실패는 결제 취소로 자동 처리하지 않는다.
- 운영자가 결제 상태와 사방넷 전송 상태를 분리해서 확인할 수 있어야 한다.
