# 의류 쇼핑몰 기능 요구사항 명세서

작성일: 2026-07-07

## 1. 목적

본 문서는 모바일 우선 의류 쇼핑몰 구축을 위한 1차 기능 요구사항을 정리한다. 쇼핑몰은 브랜드 중심 편집샵 구조를 갖고, 상품 데이터는 사방넷 API를 원천으로 사용하며, 결제는 토스페이먼츠 Standard SDK의 카드사·간편결제 결제창을 사용한다.

디자인과 화면 흐름은 `https://arena.co.kr/`의 모바일 커머스 경험을 참고한다. 레퍼런스는 화면 구조, 메뉴 흐름, 상품 탐색 방식, 콘텐츠형 커머스 구성의 참고용이며, 실제 브랜드명, 상품, 카피, 이미지, 정책은 시퀸즈 운영 기준으로 교체한다.

## 3. 산출물 범위

1차 기획 산출물은 다음 문서로 관리한다.

- 본 기능 요구사항 명세서
- 사방넷 개발용 API 문서: `docs/api/sabangnet-api.md`
- 토스페이먼츠 개발용 API 문서: `docs/api/tosspayments-api.md`

외부 서비스 문서 장애 또는 접근 지연에 대비해 서비스별 API 요약 문서를 저장소에 유지한다. 단, 외부 문서의 원문 전체를 복제하지 않고 개발에 필요한 연동 범위, 인증, 엔드포인트, 요청 흐름, 예외 처리만 정리한다. API 키, 비밀번호, 시크릿은 문서에 기록하지 않는다.

## 4. 모바일 우선 UX 요구사항

### 공통 레이아웃

- 모든 고객 화면은 모바일 화면을 기준으로 설계한다.
- 상단에는 로고, 검색, 햄버거 메뉴를 둔다.
- 하단 탭에는 홈, 찜, 히스토리, 장바구니, MY를 둔다.
- 플로팅 문의 버튼은 내부 1:1 문의로 연결한다.
- PC 화면은 1차 오픈에서 모바일 구조를 깨지 않게 표시하는 수준으로 대응한다.

### 햄버거 메뉴

- 브랜드 중심 메뉴로 구성한다.
- 좌측 또는 상단에는 브랜드, SALE, COLLECTIONS, 기획전, 리뷰, 룩북, 장바구니, 마이페이지, 검색 진입점을 둔다.
- 브랜드 선택 시 해당 브랜드의 대표 이미지, 상품군, 기획전, 룩북 연결을 보여준다.
- 브랜드와 상품군의 노출 여부와 순서는 관리자에서 변경 가능해야 한다.

## 5. 고객 화면 요구사항

### 홈

- 이미지 배너와 영상 배너를 지원한다.
- 배너는 관리자에서 등록, 수정, 숨김, 순서 변경 가능해야 한다.
- 배너 클릭 대상은 상품 목록, 상품 상세, 기획전, 컬렉션, 룩북, 외부 URL 중 선택할 수 있어야 한다.
- 홈에는 추천 상품, 기획전, 컬렉션, 룩북 섹션을 구성할 수 있어야 한다.
- 영상 배너는 외부 영상 URL과 포스터 이미지를 우선 지원한다.

### 브랜드/상품 목록

- 브랜드별 상품 목록과 전체 상품 목록을 제공한다.
- 상품 목록은 2열 모바일 그리드를 기본으로 한다.
- 상품 카드에는 이미지, 상품명, 정상가, 판매가, 할인율, 라벨, 찜 버튼, 장바구니 빠른 추가 버튼을 표시한다.
- 품절 상품 노출 여부는 관리자 정책으로 설정 가능해야 한다.
- 목록 진입 경로는 브랜드, 상품군, 검색, 기획전, 컬렉션, 찜, 히스토리로 나뉜다.

### 상품 상세

- 상품명, 브랜드, 가격, 할인, 옵션, 재고 상태, 대표 이미지, 상세 이미지 또는 상세 설명을 표시한다.
- 옵션 선택 후 장바구니 담기와 바로구매를 제공한다.
- 관련 상품, 리뷰, 배송/교환/반품 안내를 표시한다.
- 가격, 옵션, 재고는 사방넷 기준을 원칙으로 한다.
- 사방넷에서 상세 이미지나 설명을 충분히 제공하지 못할 경우 쇼핑몰 관리자에서 추가 이미지와 상세 설명만 보강할 수 있어야 한다.

### 검색/필터/정렬

- 검색 대상은 상품명, 상품코드, 브랜드, 카테고리, 색상, 스타일 태그를 포함한다.
- 추천 검색어, 최근 검색어, 인기 검색어를 제공할 수 있어야 한다.
- 필터는 브랜드, 상품군, 사이즈, 컬러, 가격, 배송, 스타일, 판매 상태를 지원한다.
- 컬러는 스와치 표시를 지원한다.
- 정렬은 신상품, 낮은 가격, 높은 가격, 인기상품, 리뷰순, 조회순을 지원한다.
- 인기상품 기준은 판매량, 조회수, 찜 수 중 운영 정책으로 확정한다.

### 콘텐츠

- 기획전, 컬렉션, 룩북은 목록과 상세 화면을 가진다.
- 기획전과 컬렉션은 연결 상품을 필수로 지원한다.
- 룩북은 이미지 갤러리와 관련 상품 연결을 선택 지원한다.
- 공개 시작일, 종료일, 노출 순서, 숨김 상태를 관리할 수 있어야 한다.
- 매거진과 SNAP은 1차 오픈 범위에서 제외하고 2차 후보로 둔다.

## 6. 회원/비회원 요구사항

### 회원가입/로그인

- 일반 회원가입과 아이디/비밀번호 로그인을 제공한다.
- 카카오와 네이버 소셜 로그인을 제공한다.
- 회원가입 시 본인인증은 필수다.
- 회원가입 필수 입력 항목은 이름, 로그인 ID 또는 이메일, 비밀번호, 휴대폰 번호, 약관 동의, 마케팅 수신 동의 여부를 포함한다.
- 소셜 로그인 후 필수 정보가 부족하면 추가 정보 입력 화면으로 이동한다.

### 비회원 구매

- 비회원도 장바구니, 주문서 작성, 토스페이먼츠 결제, 주문조회가 가능해야 한다.
- 비회원 주문조회는 주문번호와 주문자 정보 기반으로 제공한다.
- 비회원 주문에는 쿠폰, 적립금, 회원등급 혜택을 적용하지 않는다.
- 비회원이 결제 완료 후 회원가입할 경우 기존 주문 연결 여부는 2차 정책으로 둔다.

## 7. 장바구니/주문/결제 요구사항

### 장바구니

- 회원과 비회원 모두 장바구니를 사용할 수 있어야 한다.
- 회원 장바구니는 계정 기준으로 저장한다.
- 비회원 장바구니는 세션 또는 브라우저 저장소와 서버 임시 장바구니를 조합해 관리한다.
- 수량 변경, 옵션 변경, 선택 삭제, 관심상품 이동, 주문하기를 지원한다.
- 주문 전 가격, 재고, 옵션 유효성을 사방넷 기준으로 재확인해야 한다.

### 주문

- 주문서는 배송지, 주문자 정보, 상품 목록, 배송비, 할인, 적립금 사용, 최종 결제 금액을 포함한다.
- 주문 생성 시 내부 주문번호를 발급한다.
- 결제 전 주문은 `결제대기` 상태로 저장한다.
- 토스페이먼츠 승인 성공 후 내부 주문을 `결제완료`로 확정한다.
- 결제 완료 후 사방넷 대량등록용 엑셀 생성 대기 상태로 전환한다.
- 사방넷 전송 실패 시 내부 주문은 유지하고 연동 실패 상태와 재시도 기능을 제공한다.

### 토스페이먼츠 결제

- 결제 UI는 쇼핑몰 주문 화면에서 주소·혜택·최종금액·결제수단을 먼저 확정한 뒤 토스페이먼츠 카드사·간편결제 결제창을 호출한다.
- 실제 노출 결제수단은 토스페이먼츠 계약과 관리자 설정에 따른다.
- 클라이언트는 Standard SDK `requestPayment`로 선택한 카드/간편결제 결제창을 호출하고, 서버는 결제 승인 결과를 검증한다.
- 결제 성공 URL, 실패 URL, 취소 URL을 제공한다.
- `orderId`, `paymentKey`, 결제 금액을 서버에서 검증한다.
- 중복 승인, 금액 불일치, 주문 상태 불일치, 결제 실패를 방어해야 한다.
- 결제 상태 보정을 위해 웹훅 또는 결제 재조회 기능을 검토한다.

### 취소/환불

- 전체 취소와 부분 취소 지원 여부는 토스페이먼츠 계약 및 운영 정책에 따라 확정한다.
- 쿠폰, 적립금, 배송비가 포함된 주문은 취소/반품 시 재계산 규칙이 필요하다.
- 결제 취소 성공 후 내부 주문 상태와 사방넷 클레임 또는 주문 상태를 동기화해야 한다.

## 8. 쿠폰/적립금/회원등급 요구사항

### 쿠폰

- 정액, 정률, 무료배송 쿠폰을 지원한다.
- 최소 주문금액, 최대 할인금액, 브랜드 제한, 상품군 제한, 사용 기간, 중복 사용 가능 여부를 설정할 수 있어야 한다.
- 회원 전용으로 적용한다.
- 주문 취소 시 사용 쿠폰 복구 여부를 정책으로 관리한다.

### 적립금

- 구매 적립금과 리뷰 적립금을 지원한다.
- 적립 예정, 사용 가능, 사용 완료, 회수 상태를 구분한다.
- 배송 완료 후 일정 기간이 지나면 구매 적립금을 확정할 수 있어야 한다.
- 취소/반품 시 지급 또는 사용 적립금을 회수 또는 재계산한다.

### 회원등급

- 회원등급은 구매 금액 또는 주문 횟수 기준으로 산정할 수 있어야 한다.
- 등급별 할인, 적립률, 쿠폰 지급 여부를 설정할 수 있어야 한다.
- 등급 산정 기간과 등급 만료일을 관리한다.

## 9. 리뷰/문의/마이페이지 요구사항

### 리뷰

- 리뷰는 구매자만 작성 가능하다.
- 리뷰 작성 가능 시점은 배송 완료 이후로 잡는다.
- 상품별 리뷰 목록과 마이페이지 내 작성 가능 리뷰 목록을 제공한다.
- 텍스트, 평점, 이미지 첨부를 지원한다.
- 리뷰 적립금 지급 조건은 관리자 정책으로 설정한다.
- 관리자는 리뷰 노출/숨김 처리와 부적절 리뷰 관리를 할 수 있어야 한다.

### 1:1 문의

- 내부 문의 게시판을 제공한다.
- 회원은 마이페이지에서 문의 작성과 답변 확인을 할 수 있다.
- 비회원 문의 허용 여부는 운영 정책으로 확정한다.
- 관리자는 문의 목록, 상세, 답변, 상태 변경을 처리할 수 있어야 한다.
- 플로팅 문의 버튼은 문의 작성 또는 문의 안내 화면으로 연결한다.

### 마이페이지

- 회원 정보, 주문 내역, 배송 현황, 취소/교환/반품 현황을 제공한다.
- 쿠폰, 적립금, 회원등급, 주소록, 찜, 최근 본 상품, 리뷰, 문의 내역을 제공한다.
- 비회원은 주문번호와 주문자 정보로 비회원 주문조회 화면에 접근한다.

## 10. 관리자 요구사항

### 커스텀 관리자 대상

반복 사용이 많고 운영 실수가 치명적인 영역은 커스텀 관리자 화면으로 만든다.

- 주문 조회와 주문 상태 관리
- 사방넷 전송 상태와 재시도
- 결제 상태와 결제 취소 처리
- 홈 배너, 기획전, 컬렉션, 룩북 관리
- 1:1 문의 답변
- 쿠폰, 적립금, 회원등급 정책 관리
- 리뷰 노출 관리
- 배송 정책 관리
- 연동 로그와 장애 확인

### Django 기본 관리자 대상

저빈도 설정과 내부 데이터 점검은 Django 기본 관리자 기반으로 시작한다.

- 브랜드 기본 정보
- 상품군 기본 정보
- 검색어 관리
- 약관/정책 문구
- 운영자 계정과 권한
- 시스템 설정

### 사방넷 기반 상품 관리 보류 항목

상품 관리자 수정 가능 범위는 사방넷에서 제공하는 필드 확인 후 확정한다.

확인 대상:

- 상품명
- 상품코드
- 브랜드
- 카테고리
- 정상가
- 판매가
- 옵션
- 사이즈
- 컬러
- 재고
- 대표 이미지
- 상세 이미지
- 상세 설명
- 상품정보제공고시
- 판매 상태

원칙:

- 가격, 재고, 옵션은 사방넷 원천을 유지한다.
- 쇼핑몰 관리자는 노출, 순서, 추천 여부, 검색 태그, 추가 이미지, 추가 상세 설명을 보강할 수 있다.
- 사방넷 제공 범위가 충분하면 쇼핑몰 보강 필드는 최소화한다.

## 11. 사방넷 API 연동 요구사항

### 연동 원칙

- 사방넷 API는 상품, 카테고리, 주문, 운송장, 클레임, CS, 풀필먼트 범위를 검토한다.
- API 인증 정보는 환경변수 또는 배포 시크릿으로만 관리한다.
- 문서나 저장소에 계정 비밀번호, Client Secret, 토큰을 저장하지 않는다.
- API 호출은 재시도 가능하게 설계하고 모든 요청/응답 결과를 로그로 남긴다.

### 1차 필수 연동

- 상품 조회
- 카테고리 조회
- 주문 전송 또는 주문 상태 연계 방식 확인
- 운송장 조회 또는 저장 방식 확인
- 클레임 조회 또는 처리 방식 확인

### 검증 필요

- 사방넷 확인 결과 쇼핑몰 주문 신규 등록 API와 사용 가능한 웹훅은 제공되지 않는다. 주문은 시퀸즈 `[20316]` 전용 A~H 엑셀을 생성해 `주문서입력(대량)`으로 등록한다. 파일 선택과 최종 `저장`은 분리해 취급하며, 자동 업로드는 운영 검증 완료 전까지 비활성화한다.
- 상품 상세 이미지와 상세 설명 제공 범위를 테스트 상품으로 검증해야 한다.
- 재고는 사방넷 API 또는 풀필먼트 API 중 어느 범위를 사용할지 확인해야 한다.
- 배송 상태와 송장번호 입력 주체를 확정해야 한다.

## 12. 토스페이먼츠 연동 요구사항

- 토스페이먼츠 Standard SDK 결제창 연동 가이드를 기준으로 클라이언트 결제 UI를 구현한다.
- 서버는 결제 승인 API 또는 승인 결과 검증 절차를 담당한다.
- API 키는 환경변수 또는 배포 시크릿으로 관리한다.
- 결제 성공 후 내부 주문 상태, 결제 기록, 사방넷 전송 상태를 하나의 트랜잭션 흐름으로 관리한다.
- 결제 실패와 취소는 사용자에게 명확히 표시하고 내부 주문 상태를 복구 가능하게 유지한다.
- 운영 중 결제 상태 불일치가 생기면 관리자에서 재조회 또는 수동 보정할 수 있어야 한다.

## 13. 보안/장애/로그 요구사항

- 모든 외부 API 인증 정보는 저장소에 커밋하지 않는다.
- 결제 금액은 클라이언트 값을 신뢰하지 않고 서버 계산값으로 검증한다.
- 결제 승인·취소와 사방넷 엑셀 생성·등록 결과 반영은 멱등성을 고려한다.
- 외부 API 장애 시 사용자 주문이 유실되지 않아야 한다.
- 외부 API 실패는 관리자 화면에서 확인하고 재시도할 수 있어야 한다.
- API 문서 장애에 대비해 `docs/api` 아래 개발용 API 문서를 유지한다.
- API 문서는 원본 링크, 확인일, 인증 방식, 엔드포인트, 요청 흐름, 예외 처리를 포함한다.
